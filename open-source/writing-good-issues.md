# Writing Good GitHub Issues

Opening issues is one of the easiest ways to contribute to open source — and one of the most underrated.

## Issue Types

1. **Bug Reports** — Something isn't working as expected
2. **Feature Requests** — Suggesting a new feature or improvement
3. **Documentation** — Typos, unclear instructions, missing examples

## A Good Issue Template

```markdown
## Description
[What's the problem or suggestion?]

## Steps to Reproduce (for bugs)
1. Step 1
2. Step 2
3. Step 3

## Expected Behavior
[What should happen?]

## Actual Behavior
[What actually happens?]

## Environment
- OS: [e.g., macOS 15.0]
- Version: [e.g., v0.52.0]
```

## Tips

- **Search first** — Check if someone already opened a similar issue
- **Be specific** — "The docs are bad" ❌ vs "The getting started guide is missing an example for scanning Dockerfiles" ✅
- **Include context** — Why does this matter? Who benefits?
- **Be respectful** — Maintainers are often volunteers

**Key takeaway:** You don't need to write code to contribute. Good issues help maintainers prioritize work.
